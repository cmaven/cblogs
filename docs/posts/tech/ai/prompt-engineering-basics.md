# 프롬프트 엔지니어링 기초 — LLM 답변 품질을 3배 높이는 5가지 원칙

**📅 작성일**: 2026년 4월 10일

> [!NOTE]
> :bulb: LLM의 답변 품질은 프롬프트 설계에 좌우된다. 역할 부여·단계 분해·Few-shot 예시·제약 명시·포맷 지정이라는 다섯 가지 원칙을 적용하면 모호했던 응답이 실무에 바로 쓸 수 있는 결과로 바뀐다. 이 글은 ChatGPT와 Claude 모두에 적용 가능한 보편 원칙을 Before/After 예제로 정리한다.

## 프롬프트가 답변 품질을 결정하는 이유

LLM은 확률적으로 다음 토큰을 예측하는 모델이다. 같은 질문이라도 어떤 맥락과 지시를 주느냐에 따라 경로가 완전히 달라진다. "API를 빠르게 하려면?"이라는 모호한 질문과 "시니어 백엔드 엔지니어 입장에서 Node.js REST API의 p95 응답 시간 300ms를 100ms 이하로 낮추는 방법을 3단계로 설명하라"는 구체적 프롬프트는 생성되는 답변의 깊이와 정확성에서 근본적 차이를 만든다.

## 원칙 1: 역할 부여

모델의 관점과 답변 톤을 고정한다.

```
당신은 10년 차 시니어 백엔드 엔지니어다.
성능 튜닝과 프로덕션 운영 경험이 풍부하다.
```

답변이 교과서적 설명에서 실전 경험담으로 바뀐다.

> [!TIP]
> 역할은 추상적일수록 무난한 답을, 구체적일수록 날카로운 답을 얻는다. 도메인·경력·배경 지식을 섞어 쓰면 좋다.

## 원칙 2: 단계 분해

긴 작업은 step으로 쪼갠다. 모델이 중간 단계를 놓치지 않는다.

### Before

```
이 코드의 성능 문제를 분석하고 개선해줘.
```

### After

```
다음 순서로 진행하라:
1) 병목 지점 진단 (가능성 3가지 제시)
2) 각 가능성에 대한 측정 방법
3) 확정 후 개선 코드
4) 개선 전후 예상 성능 차이
```

각 단계마다 명시적 출력을 요구하면 추론 과정을 검증할 수 있다.

## 원칙 3: Few-shot 예시

입력-출력 쌍 2~3개를 보여주면 모델이 패턴을 학습한다.

```
입력: 사용자가 "오늘 날씨 어때?"라고 물음
출력: {"intent": "weather_query", "location": null, "time": "today"}

입력: "서울 내일 비 와?"
출력: {"intent": "weather_query", "location": "서울", "time": "tomorrow"}

입력: "도쿄 이번 주말 날씨"
출력: ?
```

제로샷(예시 없음) 대비 정확도가 크게 향상된다.

## 원칙 4: 제약 명시

무엇을 하지 말아야 할지도 명확히.

```
제약:
- 외부 라이브러리 사용 금지
- TypeScript로만 작성
- 함수 길이는 20줄 이하
- 주석은 한국어로
```

> [!IMPORTANT]
> "~하지 마라"는 부정 제약보다 "~만 사용하라"는 긍정 제약이 더 효과적이다. 모델이 부정문을 놓치는 경우가 종종 있다.

## 원칙 5: 포맷 지정

후처리가 필요한 경우 구조화된 포맷을 요구한다.

```
답변은 다음 JSON 스키마로:
{
  "summary": "100자 이내 요약",
  "steps": [{"order": 1, "action": "...", "code": "..."}],
  "risks": ["..."]
}
```

자동화 파이프라인에 LLM 응답을 흘려 넣으려면 포맷 지정은 필수다.

## Before / After 종합 예제

### Before (모호한 질문)

```
API 속도 개선 방법 알려줘.
```

결과: 교과서적인 일반론 (캐시, 인덱싱, CDN 등 10가지 나열).

### After (5원칙 적용)

```
역할: 당신은 대규모 트래픽 경험이 많은 시니어 백엔드 엔지니어다.

상황: Node.js + Express + PostgreSQL 기반 REST API.
      p95 응답 시간이 300ms, p99가 1.2초까지 치솟는다.
      트래픽은 초당 500 req, DB 연결풀 20.

단계:
1) 측정 지표부터 확인할 항목 3가지
2) 가장 가능성 높은 병목 지점 3가지 (우선순위 포함)
3) 각 병목별 검증 방법 (명령어 포함)
4) 1순위 병목 해결 코드 스니펫

제약: 신규 인프라 추가 없이 기존 스택 내에서 해결
포맷: 각 단계를 마크다운 헤딩 ##로 구분
```

결과: 실제 실행 가능한 진단·개선 플레이북.

## 반-패턴 (피해야 할 프롬프트)

| 반-패턴 | 왜 나쁜가 |
|---------|-----------|
| "알아서 해줘" | 모델이 평균적 답만 생성 |
| "최고의 방법은?" | 맥락 없는 베스트는 존재하지 않음 |
| 한 번에 여러 주제 질문 | 깊이가 얕아짐 — 대화를 쪼개라 |
| 결과 예시 없이 JSON 요청 | 스키마 해석이 자주 빗나감 |

> [!WARNING]
> "정확한 답을 줘"나 "환각하지 마"는 효과가 거의 없다. 대신 **출처·근거 제시를 요구**하라. 출처를 요구받으면 모델이 확신도가 낮은 주장을 스스로 걸러낸다.

## 검증 체크리스트

- [ ] 역할이 구체적인가 (직무·경력·도메인)
- [ ] 작업이 3~5단계로 분해되어 있는가
- [ ] 출력 예시나 Few-shot이 있는가
- [ ] 긍정형 제약으로 금지 영역이 명확한가
- [ ] 후처리 가능한 포맷을 지정했는가

관련 글: AI 모델 튜닝으로 서비스화하려면 [Node.js 메모리 누수 디버깅 가이드](/posts/tech/backend/nodejs-memory-leak-debugging)처럼 체계적인 측정·검증 방법론이 필요하다.

## 정리

> [!CAUTION]
> 프롬프트 엔지니어링은 만능이 아니다. 모델의 지식 경계(cutoff date) 너머의 사실이나 훈련 데이터에 없는 내부 정보는 프롬프트로 해결되지 않는다. 그런 경우에는 **RAG(Retrieval-Augmented Generation)** 나 **Tool Use**를 병행해야 한다.

프롬프트는 코드와 같다. 명확한 입력이 재현 가능한 출력을 만든다. 오늘 정리한 5원칙을 체크리스트 삼아 기존 프롬프트를 다시 점검해 보자. 응답 품질이 눈에 띄게 달라질 것이다.
